# 关于此项目

## 已改动部分
本项目基于[FutureUnreal/What-to-eat-today](https://github.com/FutureUnreal/What-to-eat-today)（2026年4月1日版本）的RAG项目，并做了以下改动。  
请注意，除了一些BUG修复外，本项目的大多改动并没有通过严格的消融实验证明一定优于原项目

### 检索路由部分
1.原项目的“查询复杂度”完全靠LLM打分  
“查询复杂度”用于测量表达形式有多复杂  
修改后：查询复杂度 = 查询长度 × 0.1 + 实体识别数量 × 0.4 + 语法结构信号 × 0.5  
其中，语法结构信号通过依存句法分析实现  

2.原项目的“关系密集度”完全靠LLM打分  
“关系密集度”用于测量是否需要利用图关系  
修改后改用以下方法测量：提取实体 → 查询实体的一度邻居“高价值节点”类型命中数量  
其中，高价值关系是由直接抽取建表语句中的“跨文档关系”得到的  
退化方案：与“预定义高关系密集度模板句”做cos相似度（取最大值）实现  

3.原项目“是否需要推理”靠LLM判断  
修改后：使用BGE+分类头的模型做判断（大概需要300条标注数据训练），显著降低了延迟、Token消耗，并增强了可解释性  

4.原项目中，用LLM判断路由选择  
修改后：有明确的硬编码的路由选择逻辑  

5.原项目中，用round robin融合文档，且使用了“先添加图RAG结果（通常质量更高）”这种假设  
修改后：使用RRF融合文档，并不偏好图RAG结果  
round robin的最大的问题是，当一个文档同时出现在两个索引中时，该方法只是普通去重而不是增加其权重  

6.原项目中，不存在对路由选择准确率的测试  
修改后：添加了测试集及测试函数，包含整个路由的测试以及BGE+分类头模型的测试  

7.并行执行以发挥最大性能优化的最大优势  

题外话：实体识别之前用spaCy的zh_core_web_sm ，但它是通用中文模型，训练数据是新闻语料，识别烹饪领域词汇极差，因此改用直接图数据库节点匹配。不过这个模型后来被用作依存句法分析了  

### 传统混合检索部分
1.原项目传统混合检索中的关键词检索使用了本地图索引  
修改后：直接调用已经初始化的BM25  
该方法看起来未完全实现完成，不仅实现略复杂，而且高度依赖“精确匹配”，而BM25至少能“词频匹配”（如果配置分词器的话），有更高的容错率。由于丢弃了图索引精确词匹配，图数据库召回兜底生效的场景变得很少，因此删除了相关操作  

2.原项目传统混合检索的每个检索方法都用到了Neo4j增强  
修改后：直接去掉了该方法  
该方法同样看起来未完全实现完成，首先是一跳邻居节点的信息很可能与“父子文本块”的设置重叠，其次是频繁调用Neo4j会引入无法忽视的延迟  

3.原项目用round robin合并关键词检索结果和向量检索结果  
修改后：使用RRF融合文档  

4.原项目文本块的metadata 里有 parent_id，但是没有返回父文档  
修改后：通过子文档回溯父文档  

5.原项目不存在重排序操作  
修改后：使用BGE Cross-encoder对RRF融合结果进行排序，并过滤掉太低的文档  

6.原项目不存在对TopK选择的测试  
修改后：添加了测试集以及测试函数  

### 图检索部分
1.原项目声明了5类查询类型，但是只实现了2个  
修改后：新增了实体关系查询、相似菜谱查询、最短路径查找（虽然和已经实现的多跳路径遍历很相似），补全了这3个查询类型的实现  

2.原项目用adaptive_query_planning作为代替路由  
修改后：由于实现太简单并且实际没有被使用（除非LLM失效）因此直接删除了。如果因为延迟原因希望启用这种硬编码路由，可以参考检索路由部分的设计思路  

### 会话缓存部分
1.“对话历史上下文”这样的短期记忆直接截取最近 3 轮对话  
修改后：滑动摘要，用查询语义相似度+时间衰减打分，取得分最高的 3 轮对话  

### 菜谱推荐部分
1.原项目推荐菜谱选项卡没有内容  
这是一个DEBUG：原先是返回字符串数组，改为前端组件期望的字典数组  

2.原项目推荐菜谱的“食材清单”、“烹饪步骤”难以准确匹配  
修改后：用LLM完成（这样会引入延迟和Token消耗，笔者能力有限，普通NLP方式似乎难以完美实现准确匹配）  

3.原项目菜谱的准备时间、推荐人数、难度、评分都是随机的  
修改后：“难度”从菜谱中直接读取，由于其他字段在菜谱中没有对应所以并没有修改  

### 后端部分  
1.原项目直接把“历史对话+新的问题”当作query交给LLM  
这是一个DEBUG：检索阶段用原始 query，生成阶段用带历史的 enhanced_query，顺带把后端日志重复的问题解决了  

### 前端部分  
1.原项目没有/cooking/[id] 路由  
这是一个DEBUG：直接在详情页内展示烹饪步骤  

### 依赖问题
原项目用Docker打包了所有依赖，本项目只使用其中的数据库容器，前后端依赖都使用手动安装  
Python依赖：pip install -r requirements.txt  
前端依赖：npm install  
这么做的原因是笔者遇到了一些网络连接和模型缓存保存问题，因此拆分开分别安装  


## 计划中的未改动部分
### 长期记忆（用户偏好、交互历史摘要等）实现
思路：  
Neo4j存储用户偏好，Milvus存储历史摘要。  
当用户请求时，用长期记忆+query作为增强query进行检索，将检索结果+增强query+短期记忆给LLM  
会话结束后，LLM提取本轮对话将长期记忆写入数据库  
检索逻辑：  
用户偏好可以按用户名过滤后全量返回（因为偏好不是和问题绑定的，而是和人绑定的）  
历史摘要则查找和当前 query 最相关的  
更新策略：  
偏好冲突：用覆盖+时间戳而不是追加（否则可能会导致新记忆和旧记忆冲突）  
摘要偏移：追加新摘要+标记过期摘要
